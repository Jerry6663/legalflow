## Phase 0 — 基础设施（14个问题，6条核心教训）

### 问题清单

| # | 类别 | 问题 | 根因 | 教训 |
|:--:|------|------|------|------|
| 1 | 认证 | GitHub PAT无法创建仓库 | Fine-grained token无`repo create`权限 | 建仓库用classic PAT |
| 2 | 认证 | Vercel CLI需token | 无cached credentials | 先让用户去网页生成token |
| 3 | 认证 | Railway只接受浏览器登录 | `railway login --token`无效 | 给用户URL+code授权 |
| 4 | 依赖 | chromadb/fastapi版本冲突 | 两个库互相锁版本 | **版本用`>=`不用`==`** |
| 5 | 依赖 | pydantic-settings启动崩溃 | 新版本pydantic不兼容 | **用纯`os.environ`替代** |
| 6 | 资源 | sentence-transformers OOM | Railway 512MB不够加载模型 | **避免重型本地模型** |
| 7 | API | DeepSeek模型名不匹配 | `deepseek-chat`实际是`v4-flash` | **先curl验证模型名** |
| 8 | 语法 | `list[str]`报错 | `from typing import list[str]`非法 | **Python 3.12用原生语法** |
| 9 | Lint | 12处flake8错误 | 尾随空格+未用导入 | **commit前本地跑flake8** |
| 10 | 导入 | `LLMClient`不存在 | `__init__.py`导出幽灵类名 | **导出与实现精确对齐** |
| 11 | 构建 | Nixpacks失败3次 | 依赖冲突+不透明 | **统一用Dockerfile** |
| 12 | Docker | CMD语法 | shell变量展开不稳定 | **用`sh -c`显式格式** |
| 13 | 网络 | Vercel被GFW拦截 | `.vercel.app`国内不可达 | **统一部署到Railway一个容器** |
| 14 | 排查 | 连环bug 7次 | 每次修一个触发下一个 | **先极简Hello World验证通路** |

### Phase 0 核心教训

1. **配置用`os.environ`，不用pydantic-settings** — 零外部依赖，永远不崩
2. **Railway用Dockerfile，不用Nixpacks** — 构建过程完全可控
3. **中国用户用Railway，不用Vercel** — GFW问题无法绕过
4. **Python 3.12用`list[str]`/`str|None`** — 不需要`typing`导入
5. **部署前本地验证语法** — `py_compile` + `flake8`
6. **先Hello World，再逐步加复杂度** — 不要同时改多个文件

## Phase 1 — 核心引擎（3个架构突破）

### W2: 三模块并行构建

| 模块 | 技术方案 | 关键决策 |
|------|----------|----------|
| 合同分类器 | 关键词预匹配(14条正则) + LLM兜底 | 劳动合同消歧：工资/社保信号优先于"开发" |
| 条款分割 | 中文条款编号正则 + LLM兜底 | 规则分出的条款≥3条才跳过LLM |
| 风险分类器 | 批量LLM调用 | 12次→1次API调用，10x速度提升 |
| RAG规则库 | ChromaDB向量(可选) + 关键词兜底 | ChromaDB在Railway上延迟加载 |

### W3: RAG法条库

- 从3个markdown文件解析77条法规
- 关键词评分匹配 + 合同类型加权
- 28个法律术语自动提取为索引

### W4: 审查Agent编排

- 7步纯Python状态机（不用langgraph外部包）
- 每个状态失败自动降级，不中断流程
- 生成44-75行结构化Markdown报告

### W5: 前端+性能优化

- 异步任务队列（`asyncio.Queue` + `Semaphore(2)`）
- 批量LLM：12次调用→1次，~43秒→~4秒
- 前端轮询 + 计时器 + 超时保护

## Phase 2 — 产品化（3个关键bug）

### Bug: 双Worker打碎内存队列

```
Worker 1: jobs = {abc: {...}}    ← submit落在这里
Worker 2: jobs = {}              ← poll查到这里，Job Not Found！
```

**修复**: 移除`--workers 2`，单worker + `asyncio`处理所有并发。

### Bug: 删代码忘记更新`__init__.py`

删除了`LLMClient`类和`AsyncOpenAI`导入，但`__init__.py`仍导出。

**教训**: 删代码时务必同步更新`__init__.py`。

### Bug: JSX嵌套被编辑打乱

多次编辑同一JSX文件导致div嵌套错位，formatter修复耗时。

**教训**: 复杂JSX编辑后用`npx vite build`验证。

## Phase 3 — 安全加固（4层防护）

### 防护层

| 层级 | 实现 | 防护类型 |
|------|------|----------|
| 网络层 | `RateLimitMiddleware` | 30次/分/IP限流 |
| 传输层 | `SecurityHeadersMiddleware` | XSS/Frame/Content-Type保护 |
| 应用层 | CORS白名单 | 只允许生产域名 |
| 数据层 | `sanitize_text` + 路径遍历防护 | XSS + 目录穿越 |

- 压测：`scripts/load_test.py`

## 架构决策树

```
技术选型
├── 框架: FastAPI (轻量+原生async)
├── AI: DeepSeek V4 Flash (性价比最优)
├── 部署: Railway单容器全栈
│   ├── 为什么不用Vercel? GFW拦截
│   ├── 为什么不用阿里云? 需备案，周期长
│   └── 为什么前后端同容器? 简化部署，避免跨域
├── 依赖策略: 最小化
│   ├── 配置: os.environ (不依赖pydantic-settings)
│   ├── 向量库: ChromaDB可选 (不在Railway加载)
│   └── 认证: SHA256+Salt标准库 (不用pyjwt)
└── 架构模式: 混合智能
    ├── 分类/拆条: 规则优先 + LLM兜底
    ├── 风险分析: 批量LLM (12次→1次)
    └── 检索: 关键词评分 + ChromaDB可选
```

## 快速维护指南

### 部署
```bash
cd frontend && VITE_API_URL=<url> npx vite build
cp -r dist ../backend/static
cd ../backend && npx @railway/cli up --detach --service legalflow
```

### 本地验证
```bash
cd backend && flake8 app --max-line-length=120
python3 -m py_compile app/**/*.py
```

### 常用Railway命令
```bash
npx @railway/cli service list           # 状态
npx @railway/cli logs --service legalflow  # 日志
npx @railway/cli variables set KEY=VAL  # 环境变量
```
